1、Workflow 和 Agentic
Agent开发中，什么时候使用 workflow，什么时候使用 Agentic，我的观点：
- 相互融合，而非替代。分别用于适当的场景。
- workflow 中的 node 可以是 agentic 的，那还是纯workflow吗？
- agentic 中也可以通过 tools 或 mcp 的方式调用 workflow，那还是纯 agentic 吗？
- workflow 适用于确定性的场景
- agentic 适用于泛用性的场景
- workflow 需要业务人员提前定义好，预置到程序里去。业务knowhow决定流程能力
- agentic 由底层大模型提供自然语言沟通能力，由tools扩展工具调用能力，由MCP扩展更丰富的交互能力，无预置程序，基于底层大模型的理解来驱动流程。大模型的理解能力和调度能力决定智能化程度。
- 我觉得正确的用法是
 - toC产品追求泛用性的应该以 agentic 为主，其中需要追求确定性的地方，比如支付，用 workflow 来实现。
 - toB产品追求确定性的应该以 workflow 为主，其中与人交互需要泛化的地方使用 agentic 来实现。
- 比较符合直觉的做法其实就是把 workflow 封装到 MCP 里，然后放到 Agentic 里。 因为 Agentic 的软件一般都是以自然语言交互为主的，不会是以流程为主，中间插入自然语言对话。

2、claudecode是一种lifestyle
如果只把 claudecode 当成编程工具，那真是屈才了。尝试了 rube 这个 MCP 服务后，世界被打开了。 rube 本身是一个MCP服务，然后它内部可启用和链接500+应用，比如twitter、github等等常见的应用。然后就可以自然语言的方式通过rube，驱动这些平台读写。
在 claudecode 安装 rube 插件后，就可以在命令行里完成和世界的交互。（不得不说，国外的平台不管API还是三方扩展基本都支持，生态好。国内好多平台连API都没有）
尝试了在 claudecode 里查询 x 上热门的AI话题，然后把分享内容发布到 x 上，它还很好得帮我做了推文拆分避免超长。
因为文本分析本身就是大模型的强项，甚至可以要求它按你的要求去别人的推下互动。集读、分析、写一体。
claudecode的用法真的可能只被发掘了5%。
从rube出发，发现还有 composio 这样的服务，也是MCP的集成服务，自己开发的 Agent 只需要对接 composio，就可以对接它背后的几百个应用，非常方便。以后开发 Agent 基本不用自己做 tools，依赖外部的 MCP 服务就行了。

3、产品思考
- 迭代，添加了多图片合成的场景，有了之前的文档积累，这次VibeCoding基本上一次过。但是遇到了上传图片尺寸超大的问题，果然网络流量才是最贵的，先用压缩图片尺寸的方式来解决。
- 价值延伸，在思考图片生成的最后一公里问题，如果只生成图片的话，价值其实不够，想知道图片生成被下载之后，还有什么进一步的用途，如果能打通最后一段的话，能提供更端到端的体验。
  - 比如：生活化的，生成图片并发送到邮箱然后触发打印机打印。商业化的，生成图片并上传到电商后台替换商品主图。社交化的，按某种规则生成头像并定时修改（发布倒计时？）这些场景都可以单独拆分出单独的工具页面来推广(搓手)。
