## 1、Cursor2.0 Composer 1 实际体验
Cursor2.0最大的更新之一就是推出了 Composer 1 自主的编码模型，号称速度最快，能力大致对标 claude-sonnet-4。
借近期的一个新项目体验了一下，结果
- 编程能力，低于 sonnet4.5 并且低于 glm4.6。
在搭建完项目框架和基础功能后，我切换到 sonnet4.5 继续开发，明显感觉到每个迭代的开发质量更好了。比如composer写出来一个前后端监听同一端口导致启动时序不稳定出现初次访问500的问题，换到sonnet4.5能一次解决好。
- 思考能力，就实际体验来说 Cursor+Composer1 时，是个沉默的编程机器，活像闷头干活的程序员。
具体比如我提出一个可能的bug原因疑问时，Composer1不会回复他的思考和决策，而是直接开始修改。Cursor的plan计划就只会展示todo-list。而切换到 ClaudeCode 时，他的回复会更有情绪，分析方案时总会提醒我的问题，而我提出疑问时，它会先回答对我疑问的判断，再展开它的计划，体验好很多。
- 定价，每天都编程的话，20刀的套餐，用不过一周就不够了。
搭建完一个基础的后台项目，3个管理功能，1个Chat智能功能，全部用Composer 1做大约用了7刀。

引申思考
- twitter上有国外工程师表示，Composer思考的时候会突然说中文，严重怀疑是从中国的开源大模型基础上训练的(qwen)，同时另外一家国外新模型也被怀疑从中国的开源模型上训练(glm)。对Cursor来说，这确实是比较优的路径。遥想当年 Deepseek V3，最初也会回答自己是ChatGPT呢。(我在文心一言上也复现过)，时过境迁啊。
- Cursor 的IDE一体化编程体验确实还不错，我判断他想把用户群锁定到零代码编程者之上，而专业程序员之下的范围，这部分选手因CursorIDE的体验而使用它是很可能的(Composer的编程能力其实也能符合这部分用户的需求，商业角度来说不是一定要追求最强)。
- 国内现在已经有 GLM 和 Kimi 两家支持包月编程套餐了(虽然也是有限额)，而且价格和网络更适合国内编程者(Curosr偶尔还是有网络问题)，所以推荐Cursor用量不够的专业选手，用 ClaudeCode + 国内编程模型的组合来AI编程。

## 2、AI编程技巧，让 Claude Code 成为代码管理专家
4种场景的最佳实践
1、ClaudeCode 自动提交代码
虽然有反对频繁提交，建议最小以完成的功能粒度来提交代码的实践。但是AI编程时，让AI细粒度提交代码，方便我们随时回滚再抽卡。
我总结的AI编程核心心法之退让原则，就是AI反复2次改不好的问题，就不要改了，回滚重做。如果未提交代码累计太多，可能同时把做好的功能也回滚，就不能使用这个方法了。以我的经验，2次改不好的问题，10次改好都有可能。甚至就改不好，导致放弃项目了。
介绍两种实现方法
- 基础实现方法，用 ClaudeCode 的 Hooks 扩展机制，在 session end，AI每次修改完成后，触发 py 或 shell 脚本完成代码提交。缺点是 message 只能固定或者按规则，不能按实际内容提交。
- 进阶实现方法，Hooks 触发 py ，连接AI，实现 message 的自动整理，按实际内容提交。缺点是 要再连一个AI，不能在 ClaudeCode 内闭环完成。

2、ClaudeCode 手动提交代码
如果不要求自动提交代码，改为手动提交，可以通过 skills 封装 git commit 的规范，利用 claudecode 来自动编写commit message并完成提交，并且 commit 的提交规范是一致的。

3、ClaudeCode 整理并推送代码
一批commit推送到远端之前，要求 ClaudeCode 整理 gitlog，用 rebase 把多个commit合并到一起(明确告诉ClaudeCode整理几次提交，或者让他自己对比远端来实现)。
不推送的时候也可以整理，时时勤拂拭。

4、从个人开发到团队协作
- 把 hook 或 skill 封装到 Plugin 里，再添加到团队私有的 marketing 里，团队成员通过私有 marketing 下载 plugin，自动安装好 hook 和 skill。(逻辑和github/gitlab共享代码基本一样，hooks依赖的py可以放到项目代码里，用相对路径触发)
