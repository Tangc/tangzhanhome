# 对ClaudeCode之父分享内容的思考 第3到13条

我是唐斩，这是第127天第146篇分享。
第143篇分享了 ClaudeCode 之父 Boris 前两条心得的思考，
今天分享其他11条的思考

## 3-模型只用最强的
boris分享他只用 opus4.5+thinking，虽然慢，但效果更好。我也有同样体验，用opus4.5重做了几个月前的一个项目，一次过的成功率高了非常多，很少有我去debug帮它查问题的时候了。

就像预算内买最好的一样，AI编程也是尽量用最强的工具+模型，
ClaudeCode+Opus4.5+Thinking 就是当前的最强AI编程组合。

最近也回答过工具和模型的选择问题，我认为可以分3个阶段
阶段1，尝试阶段，用免费的工具和模型就可以
工具推荐带GUI的
优先选Trae(选国内版，我用过，理念还不错)，qcoder、codebuddy备选(这俩我没用过)
如果是google ai的付费用户推荐 antigravity (有网络要求，和google全家桶配合最好)
模型推荐在trae里选，都免费可用。
不在trae里的话，就用qwen，有免费额度。

阶段2，正式使用，用免费的工具和不贵的付费模型
付费模型推荐用国内的
deepseek官网，api平台充值，能力顶级，价格便宜，是此阶段的不二之选
如果已经是 kimi 的会员，可以使用 kimi for coding 的编程套餐额度
api烧着心痛，更好的选择是包月套餐，推荐 glm 的 codingplan，glm4.7能力不错值得使用。
另外还有doubao的编程套餐，但实际使用的人少，还是更推荐glm
工具推荐最新的开源AI编程工具，OpenCode，此时已集成claudecode等最新的功能和实践。
因为是开源工具，未来发展靠社区肯定比较蓬勃稳定，此时开始使用正是最佳时机。

阶段3，大规模使用，用最强工具和最强模型
当世最强的竞争者没几个
ClaudeCode + Opus4.5 + Thinking 模式
Codex + GPT5.2 + xhigh 模式
antigravity + gemini 3 Pro
看到最牛的组合用法有两种
1种是 以 antigravity 为 IDE 载体，用自带的 Gemini 3 Pro 开发前端
同时开 claudecode 和 codex 的IDE插件，用来开发后端，互相配合或并行或Review
另1种是 opencode 的 oh-my-opencode 插件，同时驱动 Claude、GPT 和 Gemini 组成赛博军团完成开发任务。
土豪哥们们可以体验下，多重订阅不浪费。

## 4和5-用 CLAUDE.md 记录badcase，用 github action 记录
boris分享只用 CLAUDE.md 记录 cc 出错的地方，并且提交到git团队共享。

如果用 CLAUDE.md 记录正向规约，会非常长，很费token。反过来只记录错误就很高效，也能实现同样的错误不犯第二次的目的。
正向规约，按现在的最佳实践，更适合制作成 agent 或 skill 来实现。 保持 CLAUDE.md 精简。

团队CodeReview的时候在评论区at claude，以自动改进 CLAUDE.md ，用 gitlab 也可实现类似效果。

## 6 从 plan 模式开始
强烈推荐，总是先开 plan 模式和 CC 沟通到你没有任何问题，再让它开始执行。这条非常适合资深程序员或架构师的AI编程流程，非常适合在复杂项目中做增量开发。就像我们先选行，然后写方案，评审后再开动一样。
我曾经以为一个小改动很简单，没有plan，导致cc改了半天都没改对。最后回退了重新从plan开始，很快就改好了。
记住，plan阶段是你和AI对抗的时候，他可以拷问你，你也可以深挖它的细节，plan越细致，执行的时候就越可靠。


## 7和8 - 常用流程固化成斜杠命令或子代理
记住cc不止是编程工具，也是通用Agent。固化所有你的工作流，command，agent，skill是逐步完善的固化方式。最简单就是命令，要设定角色的就用agent，要约束结果的就用skill。
skill最近很火热，但本质都是固化CC中的工作流的方式，也是我们转变为 AI First 的重要落地方式，要重视和实践起来。
过去通过开发软件来固化数字化流程，现在通过整理 command agent skill 来固化Agent流程，是类似的逻辑但更AI First。

## 9 - 用 hooks 优化代码
boris分享cc生成90分的代码，钩子触发程序来优化最后10%。
我其实建议少用 hooks ，因为它会脱离 Agent 的上下文。 比如优化代码是可以封装skill解决的。
我读文档理解 hooks 更适合给 cc 的开发者使用，而不是使用者。
多用 agent 和 skill，少用 hooks

## 10 - 不完全放权，但预授权
深刻理解 --dangerously-skip-permissions 的危险性，文档中写的很清楚它是只适合在沙箱里使用的命令，意味着它随时可以把系统搞崩。
通过 /permissions 预授权安全的命令，是效率和安全取得平衡的好办法。
我通常是 allow command in session，这样只用授权一次，看还是要看一眼的。

## 11 - CC是通用Agent
再次强调 CC 的通用性，搭配MCP可以让CC和世界接上，帮人完成(几乎)所有事情。
我会用来搜索、调研、发推、发帖等等。
提醒一下，谨慎用MCP的超级合集，CC挑选MCP也是要费token和占用上下文的。
用什么装什么就好。
以及，理论上所有网站，只要是人可以操作的，都可以套上MCP，我已经实践过了，后续分享。

## 12 和 13 长时间运行和闭环验证
boris的最后两条都说明了，给予明确的结果和可验证的手段，CC就可以长时间运行。
前两条分享已经展示了，同时操作的Agent数量是程序员能力的衡量标准，最后两条就说明了，能让Agent运行多长时间，同样是衡量能力的标准。
你，有多长？我可能只能做到1小时。boris能做到至少1天多。

1和2条前几天我单独写了分享，提出同时操控的agent数量会成为程序员的能力衡量标准，读完最后12、13条发现，能操控agent执行多长时间的任务，也极可能成为另一项能力标准。
