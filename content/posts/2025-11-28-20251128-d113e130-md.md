# 大家好，我是唐斩，这是第113天第130篇分享。


## ClaudeCode opus4.5 和 Gemini 3 Pro 大比武
有张趣图是 OpenAI，Google，Anthropic，X，四家公司在大模型领域螺旋上升，新出的模型一定领先。
Google Gemini 3 Pro 推出后，不意外得，在多领域都获得了领先地位，特别是编程领域，把 Claude Sonnet 4.5 也比下去了。
Anthropic 也不敢示弱啊，抓紧端出了 Claude Opus 4.5 ，编程能力更强，比 opus 4.1 还便宜很多，试图站住编程能力第一的宝座。

我将用一个需求来测试一下两者能力到底如何。

### 需求
为了方便把文字分享内容发小红书，我用AI编程给自己做了一个文字切分图片的工具。
https://redimage.tangzhanx.com 免费，不需魔法。
当时为了快速实现，图片分页逻辑是用最简单的字数分，多数场景都能正常工作。
但有两种badcase
badcase 1，短列表导致一页字数不多，但是高度很大，最终文字会出页显示不全。
badcase 2，长段落导致一页字数太多，但高度不大，最终空白页面阅读体验不好。

中间快速迭代的时候尝试过改成按真实像素来切分，但是ClaudeSonnet4.0告诉我不好改，慎重。
现在我请两位高手来完成。分别使用 ClaudeCode + Claude Opus4.5 和 Cursor + Gemini 3 Pro 来实现。
使用相同的提示词，相同的流程来完成。

### 过程
按步骤展开
步骤1、描述需求
在docs文件夹中创建 prdv4.md 简述本次需求
```
## 版本需求
之前版本排版分页的核心逻辑是根据页面字数。
这样的好处：简单易实现
这样的坏处：页面内容可能超过(频繁换行时)，也可能太空白(段落字数太大时)

v4版本我们只做这一件事情，把 字数 分页，改成 真实高度 分页。
让一页能装尽可能多的内容，不多页，也不留多空白。
```

### 表现
分几个维度看
- 一次成功方面
Opus4.5一次成功运行并通过测试，Gemini3Pro遇到一个低级问题，修改一次后成功运行并通过测试。

我用 Codex + GPT-5-High 对两份代码做了评测，评测结果是
- 代码质量方面
Opus 架构更优(复用了测量模块)，并平衡了精确性和性能(极端情况可能高度不准)
Gemini3 执行更强(严格测量了高度)，但重写了侧量模块(维护难度更高)

综合结果，本次比武，Opus4.5 以微弱优势胜出。


总结，当世最强编码模型御三家都没有拉开太大的差距，可以按自己喜好来用。
因为 ClaudeCode 的超强定制能力，我最喜欢的还是 ClaudeCode 搭配 Opus4.5 用。
对于困难的任务，可以同时用三家，综合取最优结果。
模型、Agent、技师 就是 AI编程的三要素，综合最匹配才是最强。
