这两天把一个小项目部署到公网了，我本身是10多年Java后端。分享几点感想
1、技术栈选择
本着MVP最简最快，选择全js技术栈，用最现代的 nextjs + shadcn/ui 的方式。
本身也想远离自己最舒适的Java技术栈，这样能更有 vibe 感 (省得出了问题总想自己解决
找一下 AI驱动的 vibe coding 开发实践。

2、部署模式
因为运营思路是面向国内用户，后续要加商业化内容，所以最开始希望部署在国内并接支付。
那么方式就是：云平台(阿里云 或 腾讯云)，自己买服务器，买rds服务，自己搭建 CI/CD 流程，购买域名，备案(所有国内部署并提供服务的网站，规定了都要备案)。
可优化的点：把服务打成docker的image，用k8s服务一键部署。(k8s也会引入一系列的运维配置工作)。
总之是自己跑一遍运维流程。（后端做的久，基建运维自己做的少）
但是啊，如果部署在国外，选 vercel 这种web服务的 PaaS托管平台。那就是飞一般的丝滑体验了，一点都不想自己跑前面的路程。

部署国外给国内使用会有网络、账号体系、支付方式等问题，不是长久之际。
国内要跑通完成商业化流程的话，基本上是需要公司的，微信登录、支付都只能对公接。


3、js vs Java ( maybe 独立开发VS企业级开发 )
在被 lint(js的语法检查) 问题反复折磨的时候，我又想起了 Java 静态编译检查的好。
但js确实是从0-1更适配 vibe coding 的选择 (
- 代码量更小，节省LLM上下文
- 脚本语言更灵活，更适合 LLM 发挥
- LLM 的训练语料更足，主流技术栈都好使
)
独立开发、全栈开发，从前端起步的人更多，做面向用户的产品体验更适合(挣泛C的钱一定要从产品体验入手)
企业级开发，从dephi、c++、c#等沿革下来的Java开发人员更多，对并发、数据等的处理Java更适合(现在的 go 等也不错)，做面向企业的产品更适合(不一定好看，但要够稳，性能吞吐能能做到

Java似乎没有往web开发更进一步的动力，从技术栈上落后js体系太多(nextjs等开箱即用，springboot也只是追赶而已，差很多)。
那从 托管平台到技术框架，让 Java 开发者也能丝滑体验，是否是一个技术价值点呢？🤔
