# 大家好，我是唐斩，这是第92天第109篇分享。

## 1、Langchain推翻重写，涅槃重生1.0
使用最广泛的Agent开发框架 langchain 和 langgraph，推出 1.0 版本了。
为了这个里程碑版本，开发团队号称 “推翻重写” 了 langchain 框架。

langchain最初是为解决和LLM对接而产生的，随着LLM 和 Agent的急速发展，langchain里也堆积了非常多的代码逻辑。后来为了能开发更复杂架构的 Agent，单独推出了 langgraph，基于图和ReAct的方式来实现 Agent。

我之前一直以为，langgraph 是新出的框架，那么 langgraph 应该是依赖 langchain 来完成和 LLM 对接的。甚至 Spring AI Alibaba 的介绍里有说自己是对标的 langgraph，而其依赖的 Spring AI，对标 langchain。现在就很尴尬了。

这次重写后的架构是，langchain 作为高级封装，底层依赖 langgraph ，可以快速开箱即用和 LLM 对接上。langgraph作为提供低级接口，可高度自定义 Agent 的行为。

langgraph1.0定义了标准的和LLM交互的模型， request -> model -> tools -> result 
同时提供了一种叫 middleware 中间件的抽象层，可以在model前后，agent前后插入自定义逻辑，实现诸如人机交互，数据加密，结构化输出等逻辑。

下次新项目就用1.0来开发，深度体验一下(虽然都是AI写^ ^)

## 2、我还应不应该写长文
思考一个问题，我还应不应该写长文？
现在的方式是写分享(1000字左右)，发布到小红书，然后制作成视频脚本拍成短视频发布到多个平台。
最近看到一个公众号每篇标题都是"万字详解xx"，"两万字解析xx"一个技术主题，内容有深度有广度，值得阅读。
当我自己想写长文时发现长文写作比每日短文写起来难非常多。
所以产生了这么一个问题。

思考的结论是两点。
1. 应该写长文。长文更难产出，更有阅读价值，对专业内容的用户更有吸引力，也是做出下一阶深度内容的前提。
2. 输出内容和发布渠道应该分开。内容是基础，有内容后，发布渠道是以内容会友的一种方式，不发布线下讲也是方式。

尝试以此结论构思一套内容和发布体系，并实践。
- 内容体系，从短到长，从碎片到完整，价值从低到高，深度从浅显到深入，受众从广泛到精准，壁垒从低到高。
  - 碎片，百字。一个观点，一个判断，一个灵感，一个思考方向。短小，作为后续内容的入口。比如 上下文工程核心之一是上下文的数据结构设计。
  - 分享，千字。在碎片基础上拓展成一篇完整的内容，有核心论点和论述，少量举例和实践。比如 上下文的数据结构设计方法，一般由 memory，systemprompt，fewshot ... 构成，出处是 Anthropic
  - 图文，2-3千字+图解。在分享基础上完善，主要是补充实践。比如针对AI导购Agent的上下文数据结构设计的案例。
  - 短视频，基于分享或者图文制作，但要改写成脚本再拍。
  - 长文，万字以上。内容结构大概是理论+调研+实践+应用多方面综合，应该要对一个主题有时间或者空间的论述总结。比如 上下文结构设计的理论，开源Agent项目的上下文结构横向对比，实际测试结果和我应用到项目中的案例效果。待实际写一写。
  - 长视频，待探索。
  - 体系文，同一主题方向上的一系列长文组合而成。比如上下文工程，包括结构设计、内容压缩等等。待实践。
  - 课程，在体系的基础上搭建，待探索
  - 实战，在课程的基础上搭建，待探索
- 发布体系
  - 碎片，不发布
  - 分享，架构师之路星球，首发。

  - 短视频，视频号、小红书、抖音、快手、B站
